0.  Architecture Overview & Design Principles 
This document specifies the complete MongoDB NoSQL schema for the RiskHub platform. Four 
collections form the core data layer, each designed for a specific access pattern driven by the 
Behavioral Quant Engine, the real-time alert pipeline, and the SBT (Soulbound Token) Oracle workflow. 
 
Core Design Philosophy 
Pattern-First Design: Every field exists because a specific query or computation requires it. No 
speculative fields. Every index has a named query pattern it serves. Denormalization is deliberate — 
we embed data that is always read together to eliminate JOIN-equivalent $lookup operations on hot 
paths.

0.2  Global Design Decisions 
• ObjectId for _id: MongoDB default ObjectId is used throughout. It encodes a timestamp, 
enabling time-based document ordering as a free secondary property. 
• Decimal128 for financial values: All monetary amounts, percentages, and ratios use 
NumberDecimal (Decimal128) to avoid IEEE 754 floating-point rounding errors in PnL 
calculations. 
• ISODate for all timestamps: UTC-normalized ISODate is used for every timestamp. 
Application layer converts to user timezone for display only. 
• AES-256-GCM encryption at application layer: Exchange API keys are encrypted by the 
FastAPI backend before any write to MongoDB. The database never holds plaintext credentials. 
The envelope format is enc::iv::tag::ciphertext. 
• Compound indexes lead with user_id: All multi-field indexes place user_id as the first 
component. This is the primary equality filter for all queries, maximizing index selectivity. 
• TTL indexes for automatic data hygiene: Ephemeral data (SIWE nonces, old alert records, 
stale metrics) is automatically purged by MongoDB TTL indexes, eliminating scheduled cleanup 
jobs. 
• Schema versioning: Every document includes schema_version: 1. Future migrations can 
target specific versions in bulk update scripts without full collection scans. 
 
 
1.  Collection: users 
1.1  Purpose 
The users collection is the identity root of the entire platform. It stores authentication credentials, linked 
Web3 wallet addresses, encrypted exchange API key pairs (the most security-sensitive data in the 
system), and the user's current Soulbound Token minting state. All other collections reference users._id 
as their primary partition key.

1.4  Design Rationale 
Embedded Exchange Keys Array 

RiskHub — MongoDB Schema Design Database Architecture v1.0 
`© 2025 RiskHub. Confidential. Page 5 
Exchange API keys are stored as an embedded array inside the user document (exchange_keys[]) 
rather than in a separate collection. This is a deliberate denormalization choice: the frontend dashboard 
always needs both user profile data and exchange connection status in a single read. Embedding 
eliminates a $lookup. The array is bounded in size (users typically connect 2–5 exchanges), so it never 
causes document growth issues. 
AES-256-GCM Encryption Envelope 
SECURITY CRITICAL — Application-Layer Encryption 
Fields api_key_encrypted, api_secret_encrypted, and passphrase_encrypted are NEVER stored in 
plaintext. The FastAPI backend encrypts them using AES-256-GCM before any database write. 
Format: enc::<base64_iv>::<base64_tag>::<base64_ciphertext>. The master encryption key is 
stored only in environment secrets (AWS Secrets Manager / HashiCorp Vault). If the MongoDB 
database is compromised, encrypted key blobs are useless without the master key. Raw API keys 
are zeroed from memory immediately after encryption. 
 
SBT Subdocument — Embedded vs Reference 
The SBT status object is embedded in the user document rather than in a separate collection because: 
(1) there is exactly one SBT per user (1:1 relationship); (2) the dashboard always reads SBT status 
alongside user profile data; (3) atomic updates to both user data and SBT status happen in a single 
document operation, which is trivially atomic in MongoDB without transactions. 
Sparse Index on wallet.address 
The unique index on wallet.address uses sparse: true because wallet connection is optional — users 
can use RiskHub with email/password only. A non-sparse unique index would prevent more than one 
user from having a null wallet address, which would break email-only registration. A sparse index 
ignores null/absent fields entirely. 
 
 
2.  Collection: trade_history 
2.1  Purpose 
The trade_history collection is the largest and most write-intensive collection in RiskHub. It stores every 
closed trade position fetched from every connected exchange for every user. It is the primary input to 
the Behavioral Quant Engine — all behavioral rules (Revenge Trading, Overtrading, Leverage 
Escalation), the Discipline Score, Win Rate, and Max Drawdown are computed by querying this 
collection. The schema is optimized for high-volume time-range scans sorted by recency.

2.4  Design Rationale 
Compound Index: { user_id, exchange_id, closed_at: -1 } — The Primary Workhorse 
This is the single most important index in the entire schema. The Quant Engine runs on a 60-second 
polling cycle and must efficiently fetch the last 30 days of trades for each user-exchange pair. The 
index structure follows the ESR rule (Equality-Sort-Range): user_id and exchange_id are equality filters 
(highest selectivity first), and closed_at is the sort/range key. This allows MongoDB to satisfy the query 
entirely from the index B-tree with a bounded range scan and no in-memory sort. 

RiskHub — MongoDB Schema Design Database Architecture v1.0 
`© 2025 RiskHub. Confidential. Page 8 
Deduplication Index on exchange_trade_id 
The 60-second polling cycle will repeatedly fetch overlapping trade windows from exchange APIs. 
Without a deduplication guard, the collection would accumulate duplicate records, corrupting all 
aggregate calculations. The unique compound index on { user_id, exchange_id, exchange_trade_id } 
causes MongoDB to reject duplicate inserts with a DuplicateKey error, which the backend handles 
silently (upsert pattern). This is cheaper and more reliable than application-level deduplication. 
Decimal128 for PnL Fields 
Why Not Float64 (Number)? 
Standard JavaScript/Python float64 (IEEE 754 double) cannot represent many decimal fractions 
exactly. For example: 0.1 + 0.2 == 0.30000000000000004 in float64. When accumulated across 
thousands of trades, these rounding errors corrupt PnL totals and Discipline Score components. 
MongoDB's Decimal128 type uses base-10 arithmetic with 34 significant digits — matching the 
precision of financial exchange data. All monetary fields (realized_pnl_usd, entry_price, etc.) must be 
written as NumberDecimal(). 
 
Denormalized is_win Boolean 
The is_win field is a derived value (realized_pnl_usd > 0) that could be computed on the fly. It is 
deliberately stored as a denormalized field because the consecutive-loss-streak rule (BHV-005) 
requires a sorted scan filtered by is_win = false. Without the stored field, this would require a $expr 
operator on a Decimal128 comparison, which cannot use a standard index. The stored boolean allows 
a fast covered index scan. 
raw_exchange_data Subdocument 
The raw CCXT response is embedded for auditability and debugging. If a user disputes a calculated 
metric, engineers can inspect the original exchange payload without re-fetching. In production, this field 
can be moved to a cold-storage collection after 90 days if storage costs become a concern. 
Sharding Strategy (Phase 2) 
At scale (50M+ documents), the trade_history collection should be sharded on { user_id: 'hashed' }. 
Hashed sharding ensures even distribution of user data across shards and keeps all trades for a single 
user on the same shard (zone sharding optional for data residency). The primary workhorse index 
remains efficient within each shard. 
 
 
3.  Collection: risk_metrics 
3.1  Purpose 
The risk_metrics collection stores point-in-time snapshots of all computed Quant Engine outputs after 
every sync cycle. It serves two distinct read patterns: (1) fetching the single latest snapshot for 
dashboard rendering (low-latency point lookup), and (2) fetching a time-range of snapshots for drawing 
historical charts (Discipline Score trend, Max Drawdown history). It is also the source of truth for the 
SBT Oracle — the signed payload is derived from the most recent snapshot.

3.4  Design Rationale 
Append-Only Time-Series Log — No In-Place Updates 
Every sync cycle creates a new document rather than updating an existing one. This is a fundamental 
time-series design principle: immutable append-only records. Benefits: (1) Full audit trail of how metrics 
evolved; (2) Chart data is a natural time-range query with no aggregation pipeline needed; (3) No write 
conflicts on the latest document from concurrent processes; (4) Rollback is trivial — just query a 
previous snapshot. The trade-off is storage growth, handled by the 1-year TTL index. 
Covered Index for Chart Queries 
Performance Pattern: Covered Index 
The chart API endpoint needs to return (calculated_at, discipline_score.total, 
max_drawdown.value_pct, win_rate.value_pct) for up to 90 data points. The compound index 
idx_user_chart_covered includes all these fields. MongoDB can satisfy this query entirely from the 
index without loading the full documents from disk (a covered query). For a dashboard serving 1,000 
concurrent users, this eliminates ~90% of disk I/O on the most frequent read endpoint. 
 
by_exchange Denormalized Array 
Per-exchange metric breakdowns are embedded as an array rather than as separate documents in a 
sub-collection. The dashboard per-exchange panel always reads all exchange breakdowns together 
(never one at a time), so embedding avoids N+1 query patterns. The array size is bounded by the 
number of connected exchanges (typically 2–5). 
SBT Payload Preparation 
The sbt_payload_hash field stores the keccak256 hash of the metrics payload at calculation time. 
When the user initiates SBT minting, the backend fetches the latest snapshot, verifies its hash (integrity 
check), signs it with the Oracle private key, and returns the signed payload to the frontend. Storing the

latency. 
MongoDB Native Time-Series Collection (Future Optimization) 
MongoDB 5.0+ offers a native Time Series Collection type with automatic bucket compression,

2, risk_metrics should be migrated to a native time-series collection with timeField: 'calculated_at' and 
metaField: 'user_id'. This can reduce storage by 50–80% and improve range query performance 
further. 
 
 
4.  Collection: alerts_log 
4.1  Purpose 
The alerts_log collection stores every behavioral alert event generated by the Quant Engine. It must 
support three concurrent access patterns: (1) real-time alert delivery status tracking (was the 
WebSocket push successful?), (2) fast unread-count queries for the notification badge UI, and (3) 
paginated alert history with filtering by severity and rule type. The rate-limiting logic also queries this 
collection to prevent alert fatigue.

4.4  Design Rationale 
trigger_context Subdocument — Embedded Evidence 
The trigger_context object embeds a snapshot of the exact data that fired the rule: the loss amount, the 
triggering trade's leverage, the time delta, and the threshold values. This is critical for two reasons: (1) 
the alert message can be rendered with precise, user-specific data (e.g., 'You opened a 20x position 6 
minutes after a $72.50 loss') rather than a generic string; (2) if the trade_history records are later 
modified or purged, the alert evidence remains intact for audit purposes. 
Rate Limiting via rate_limit_key Index

`© 2025 RiskHub. Confidential. Page 15 
Alert Fatigue Prevention Design 
The Quant Engine evaluates rules on every 60-second sync cycle. Without rate limiting, a persistent 
revenge-trading pattern could generate 60 alerts per hour. The rate_limit_key field is a composite 
string (rule_id + user_id) indexed for fast point lookup. Before firing any alert, the engine queries: 
findOne({ rate_limit_key: '...', triggered_at: { $gte: 30-min-ago } }). If a document exists, the alert is 
suppressed. This check adds ~1ms latency per rule evaluation and prevents alert storms. 
 
related_trade_ids as ObjectId Array 
The alert stores references (ObjectId) to the specific trades in trade_history that caused the rule to fire. 
This enables the frontend alert detail view to deep-link directly to the relevant trades, giving users 
actionable context. Array references are preferred over embedding full trade documents because trade 
records can be updated (e.g., fee reconciliation), and embedded copies would become stale. 
expires_at TTL vs Static TTL Index 
Rather than using a static expireAfterSeconds on created_at (which would delete all documents older 
than N seconds from creation), the schema uses an explicit expires_at field with expireAfterSeconds: 0. 
This means MongoDB deletes the document when the current time passes expires_at. This approach 
allows future flexibility: high-severity alerts (critical liquidation warnings) could be given a 90-day 
retention period by writing a farther-future expires_at value, while routine notices expire in 30 days — 
all served by the same index.

6.  Query Performance Guidelines for Backend 
Engineers 
6.1  Always Use hint() in Quant Engine Queries 
The Quant Engine runs on a tight 60-second cycle. MongoDB's query planner can sometimes choose a 
suboptimal index on complex queries. Explicitly hint the index name on all Quant Engine queries (as 
shown in the code blocks above) to guarantee plan stability and prevent unexpected performance 
degradation after collection statistics change. 
6.2  Use Projection to Minimize Document Transfer 
When building the dashboard, always specify a projection object to return only required fields. For 
example, the risk summary widget needs only discipline_score.total, max_drawdown.value_pct, 

RiskHub — MongoDB Schema Design Database Architecture v1.0 
`© 2025 RiskHub. Confidential. Page 17 
win_rate.value_pct, and calculated_at — not the full 40-field document. Projection reduces network 
transfer, deserialisation time, and application memory usage. 
6.3  Prefer Upsert over findOne + insert for Trade Ingestion 
The trade ingestion loop should use bulkWrite() with updateOne + upsert:true operations, keyed on the 
deduplication index fields. This is a single round-trip to MongoDB regardless of batch size, compared to 
N individual insert calls. Recommended batch size: 500 documents per bulkWrite call. The 
deduplication unique index guarantees idempotency automatically. 
6.4  Connection Pooling 
The FastAPI backend uses Motor (async MongoDB driver). Configure the connection pool with 
minPoolSize: 5, maxPoolSize: 50 for the MVP. The Quant Engine's parallel exchange polling creates 
bursts of concurrent database writes — the pool prevents connection exhaustion. Monitor the pool 
utilization metric in MongoDB Atlas and increase maxPoolSize if avg wait time exceeds 10ms.
