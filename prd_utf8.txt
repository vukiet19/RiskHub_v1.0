1. Executive Summary 
RiskHub is a proactive, read-only Web3 portfolio risk management platform designed to serve 
crypto investors operating across multiple centralized exchanges. In a market defined by emotional 
volatility, opaque performance claims, and fragmented asset visibility, RiskHub positions itself not 
as a trading tool, but as a behavioral and analytical shield — a platform that observes, quantifies, 
and warns. 
 
The platform solves three compounding problems that disproportionately harm retail traders: (1) the 
inability to see one's full cross-exchange risk exposure in a single view; (2) the lack of an objective 
feedback mechanism to identify and interrupt emotional trading behaviors; and (3) the absence of 
tamper-proof, verifiable on-chain credentials for trading performance, which enables fraudulent 
copy-trade schemes. 
 
RiskHub addresses each of these through four integrated core capabilities: cross-exchange read-
only data ingestion via the CCXT library, a real-time Python-based Behavioral Quant Engine, an 
interactive Asset Contagion Graph, and an immutable Web3 Soulbound Token (SBT) that 
permanently encodes a user's verified trading discipline on-chain. 
 
The MVP scope targets active crypto traders on Binance and OKX, delivering a functional 
dashboard, behavioral alerting system, and SBT minting on BNB Chain Testnet within a 16-week 
build cycle.

2. Problem Statement 
The cryptocurrency trading ecosystem imposes asymmetric information disadvantages on retail 
participants. Three structural problems converge to create an environment where retail traders 
consistently underperform and are actively exploited. 
 
2.1  Fragmented Cross-Exchange Portfolio Visibility 
Professional and retail traders alike routinely operate across 3–10 exchanges simultaneously to 
access diverse liquidity pools, hedging instruments, and yield products. No native tooling 
aggregates Spot and Futures balances — including unrealized PnL, locked margin, and liquidation 
thresholds — into a single, coherent risk view. This fragmentation systematically conceals true 
portfolio exposure and prevents rational risk-adjusted decision-making. 
 
2.2  Behavioral Undiscipline and Emotional Trading 
Quantitative studies of retail trading outcomes consistently identify behavioral patterns — not 
market conditions — as the primary cause of account drawdowns. Patterns such as Revenge 
Trading (immediately increasing leverage following a loss), Overtrading (high-frequency position 
entry without edge), and Averaging Down into losing positions are broadly documented. No existing 
retail platform provides a real-time feedback mechanism that detects and interrupts these patterns 
as they form. The gap between knowing a behavior is harmful and receiving an objective, in-context 
warning is the precise space RiskHub occupies. 
 
2.3  Absence of Verifiable Trader Identity and Performance Records 
The copy-trading and signal-selling industry is riddled with fraud. Manipulated PnL screenshots, 
selective track records, and fabricated win-rate statistics are standard practice among fraudulent 
'crypto gurus'. Platforms that facilitate copy-trading have no reliable mechanism to verify the true 
historical risk-adjusted performance of a signal provider. This information asymmetry causes 
measurable financial harm to thousands of followers who allocate capital to unverified traders. The 
solution requires a tamper-proof, publicly queryable, and immutable record of a trader's true 
behavioral discipline and performance metrics — one that cannot be cherry-picked, edited, or 
deleted.

5. System Architecture 
5.1  Architecture Overview

extensibility. Each layer has a defined responsibility boundary and communicates with adjacent

5.2  Data Flow — Read-Only Ingestion Pipeline 
User provides read-only API key and secret via the frontend. Credentials are encrypted (AES-256) 
before storage in MongoDB. The FastAPI backend invokes the CCXT library to fetch balances, 
positions, and trade history from the target exchange. Fetched data is normalized to a unified 
schema and stored in MongoDB. A background Celery task re-fetches data on a configurable 
schedule (default: every 60 seconds for live positions, every 5 minutes for historical trade data). 
Normalized data is published to a Redis pub/sub channel, which the WebSocket server subscribes 
to, pushing updates to connected frontend clients. 
 
5.3  Data Flow — Quant Engine & Alert Pipeline 
Upon each data update, the Quant Engine is invoked as a Celery task. The engine reads trade 
history from MongoDB and applies the full suite of heuristic rule evaluators (see Section 6.2). Any 
triggered rules generate Alert objects, which are persisted to MongoDB and simultaneously pushed 
via WebSocket to the connected frontend client. The frontend renders alerts as dynamic, non-
blocking overlay notifications.

5.4  Data Flow — SBT Minting Pipeline 
User initiates SBT mint from the frontend. The backend aggregates the user's complete risk metrics 
payload (drawdown, win rate, leverage profile, discipline score) and signs it using a private key held 
by the RiskHub Oracle service (Web3.py). The signed payload and signature are returned to the 
frontend. The user's MetaMask wallet (connected via Wagmi + RainbowKit) submits a transaction to 
the RiskHubSBT smart contract, calling the mint function with the signed payload. The smart 
contract verifies the Oracle's signature, creates a non-transferable SBT bound to the user's wallet 
address, and emits a SBTMinted event. The token is publicly queryable on-chain by any third party. 
 
5.5  Security Architecture 
• Read-Only Enforcement: CCXT library is configured exclusively with read-scope API keys. 
No trade execution, withdrawal, or fund transfer methods are called or exposed. 
• Credential Security: Exchange API keys are encrypted at rest using AES-256 symmetric 
encryption. Encryption keys are managed via environment secrets (not stored in the 
database). 
• Oracle Key Security: The RiskHub Oracle signing key is stored in a Hardware Security 
Module (HSM) or secure key vault in production. The key is never exposed to the client 
layer. 
• Authentication: All backend API endpoints require JWT authentication. Web3 wallet 
signature (Sign-In With Ethereum / Sign-In With BNB) is supported as an alternative auth 
mechanism. 
• Network Security: All client-server communication over HTTPS/WSS. CORS restricted to 
known frontend origin domains.

6.1  Feature: Read-Only Cross-Exchange Data Ingestion 
Priority: P0 — MVP Critical 
 
6.1.1  User Story 
As an active trader, I want to connect my exchange accounts using read-only API keys so that 
RiskHub can aggregate my portfolio data without ever having access to execute trades or withdraw 
funds. 
 
6.1.2  Functional Specifications 
• The system shall support API key connection for Binance and OKX in the MVP, with a 
framework extensible to all 100+ CCXT-supported exchanges in future phases. 
• The API key connection interface shall clearly label each permission scope (Read-Only) and 
block any attempt to store keys with write or withdrawal permissions. 
• The system shall fetch and unify the following data points per connected exchange: total 
account balance (USDT equivalent), Spot holdings (asset, quantity, current value), active 
Futures positions (symbol, side, entry price, current price, leverage, unrealized PnL, 
liquidation price, margin used), and historical closed trades (timestamp, symbol, side, 
quantity, entry price, exit price, realized PnL, leverage used). 
• All data from multiple exchanges shall be normalized to a single internal schema and 
aggregated into a unified portfolio view, denominated in USDT. 
• The system shall re-sync live position data every 60 seconds and historical trade data every 
5 minutes. 
• If an exchange API key becomes invalid or returns an error, the system shall surface a non-
blocking warning notification to the user without crashing the dashboard. 
 
6.1.3  MVP Acceptance Criteria 
1. A user can successfully add a Binance read-only API key and see their Spot and Futures 
portfolio data rendered on the dashboard within 10 seconds of connection. 
2. A user can successfully add an OKX read-only API key, and data from both exchanges is 
aggregated and displayed in a single unified balance view. 
3. The system is demonstrably incapable of placing any trade or withdrawal on a connected 
exchange account (verified via CCXT configuration audit). 
4. An invalid or expired API key triggers a user-visible warning badge on the exchange 
connection panel within one refresh cycle.

6.2.3  Calculated Metrics 
• Max Drawdown: Calculated as ((Peak Portfolio Value - Trough Portfolio Value) / Peak 
Portfolio Value) * 100, computed across the full available historical trade window. 
• Trading Discipline Score (TDS): A composite score (0–100) derived from the inverse 
frequency of triggered behavioral rule violations, normalized by trading volume. A score of 
100 indicates zero behavioral violations. Score components are: Leverage Control (30%), 
Drawdown Management (30%), Trade Frequency Discipline (20%), Loss Recovery Behavior 
(20%). 
• Win Rate: (Number of profitable closed trades / Total closed trades) * 100. 
• Average Leverage: Mean leverage across all closed Futures positions in the trailing 30-day 
window. 
 
6.2.4  Alert Delivery Requirements 
• Alerts shall be delivered as dynamic overlay pop-up cards in the bottom-right of the 
dashboard, persisting until acknowledged by the user. 
• Each alert card shall display: severity icon (color-coded: red = high, orange = medium, 
yellow = info), alert title, detailed description message, triggered timestamp, and an action 
button (Acknowledge / View Details). 
• Acknowledged alerts are archived in a persistent Alert History panel accessible from the 
dashboard sidebar. 
• Alerts shall be delivered via WebSocket push; no polling mechanism shall be used for alert 
delivery. 
 
6.2.5  MVP Acceptance Criteria 
6. The Quant Engine correctly detects a simulated Revenge Trading event (RQ-001) and

event. 
7. The Trading Discipline Score renders correctly on the dashboard and updates after each 
data sync cycle. 
8. Max Drawdown is calculated correctly for a test dataset and matches a pre-verified expected 
output (within 0.01% tolerance). 
9. Alert cards render correctly, persist until acknowledged, and are archived in the Alert History 
panel upon acknowledgement. 
10. All five MVP behavioral rules (RQ-001 through RQ-005) are implemented and unit-tested 
with pass/fail test cases.

6.3  Feature: Asset Contagion Graph 
Priority: P1 — MVP Important 
 
6.3.1  User Story 
As a portfolio manager, I want to visually understand the correlations between my held assets so 
that I can identify when my portfolio is effectively concentrated in a single risk factor and proactively 
reduce cascading liquidation exposure. 
 
6.3.2  Functional Specifications 
• The Contagion Graph shall be rendered using React Flow as an interactive, zoomable, and 
pannable network visualization embedded on the main dashboard. 
• Each asset in the user's portfolio shall be represented as a node. Node size shall be 
proportional to the asset's weight as a percentage of total portfolio value. 
• Edges between nodes shall represent pairwise Pearson correlation coefficients calculated 
from 30-day rolling daily returns. Edge thickness shall be proportional to correlation 
magnitude (|r|). Edges with |r| < 0.3 shall not be displayed to reduce visual noise. 
• Node and edge color coding: Correlation >= 0.7 (high): Red edge; Correlation 0.4–0.69 
(moderate): Orange edge; Correlation < 0.4 (low): Grey edge. If a node's asset drops more 
than 5% in a single day, the node and all connected edges with |r| >= 0.5 shall pulse red to 
indicate contagion risk. 
• Hovering over any node shall display a tooltip showing: asset name, current value, portfolio 
weight (%), and a list of the top 3 most correlated assets with their correlation coefficient 
values. 
• A Contagion Risk Score shall be calculated as the average correlation of the top-3 asset 
pairs by portfolio weight. This scalar score (0–100) shall be displayed prominently above the 
graph. 
 
6.3.3  MVP Acceptance Criteria 
11. The graph correctly renders nodes for all assets in a test portfolio with more than 3 holdings. 
12. Node sizes are proportionally correct (verified by visual inspection against known portfolio 
weights). 
13. Correlation values are computed correctly against a pre-verified reference dataset (within 
0.02 tolerance). 
14. A simulated 6% single-day drop in BTC triggers the visual contagion pulse effect on all BTC-
adjacent nodes with correlation >= 0.5. 
15. The graph is interactive: zoom, pan, and hover tooltip all function correctly in Chromium-
based browsers.

13. Glossary of Terms 
Term Definition 
CCXT CryptoCurrency eXchange Trading Library. An open-source library 
providing a unified API interface to 100+ cryptocurrency exchanges. 
SBT (Soulbound 
Token) 
A non-transferable, non-tradeable blockchain token permanently bound to a 
specific wallet address, representing verifiable credentials or identity 
attributes. 
Max Drawdown The maximum observed percentage decline in portfolio value from a 
historical peak to a subsequent trough, before a new peak is achieved. 
Revenge Trading The behavioral pattern where a trader immediately re-enters the market 
with increased leverage or position size following a significant loss, driven 
by the emotional desire to recover losses quickly. 
Oracle In the Web3 context, a trusted off-chain service that fetches real-world data, 
signs it cryptographically, and delivers it to a smart contract in a verifiable 
format. 
Contagion Risk The risk that a decline in one asset triggers correlated declines in other held 
assets due to high cross-asset correlation, amplifying total portfolio losses 
beyond what any single position would suggest. 
Trading Discipline 
Score (TDS) 
A proprietary composite metric (0–100) calculated by RiskHub's Quant 
Engine reflecting the consistency of a trader's adherence to risk 
management best practices over time. 
ERC-5114 An Ethereum Improvement Proposal standard defining the interface for 
Soulbound tokens — non-transferable NFTs bound to a specific EOA or 
smart contract wallet. 
P95 Latency The 95th percentile response time. 95% of all requests are completed 
within this time threshold. A standard metric for measuring system 
performance under real-world load. 
Read-Only API Key An exchange API credential with permissions restricted exclusively to 
reading account data (balances, positions, trade history). Cannot execute 
trades, place orders, or trigger withdrawals.
